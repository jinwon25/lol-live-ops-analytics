# CLAUDE.md — 컴투스 데이터분석 포트폴리오

이 문서는 본 프로젝트의 **단일 진실의 원천(single source of truth)** 입니다.
앞으로 모든 작업(코드, SQL, 문서, 커밋 메시지)은 이 문서의 원칙을 따릅니다.

---

## 1. 목표 (Goal)

게임 회사(**모바일 가챠 RPG, 라이브 운영**) 데이터 분석 직무 지원용 포트폴리오.
롤(LoL) 랭크 데이터를 활용하되, **"누가 이기나" 같은 승부 예측은 절대 하지 않는다.**

**라이브 게임 분석가의 시선**으로 다음 3개 축을 분석한다.

| 축 | 주제 | 결과물 |
|---|---|---|
| **축1** | 메타 건강도 + 패치 드리프트 | 챔피언 픽률/승률, 다양성 지표(HHI/지니), 너프·버프 후보 자동 플래깅, **패치 15.1 → 15.3 변화 추적** |
| **축2** | 유저 행동 세그멘테이션 | 플레이어를 행동 유형으로 **K-means 군집화** + 페르소나 정의 |
| **보조** | 데이터 수집 파이프라인 | **Riot API**로 직접 수집하는 코드 (역량 증명용, 소량이라도 실 작동) |

## 2. 핵심 제약 (Hard Constraints)

> 이 제약을 어기면 분석 자체의 신뢰성이 무너집니다. 모든 산출물은 이 규칙을 통과해야 합니다.

1. **지역**: 입력은 두 파일 모두 **EUN(Europe Nordic & East)** 지역. 다른 지역으로 일반화하지 않는다.
2. **시점 비교**: 두 파일 모두 **2025년 초** 시점이며 약 2패치 차이. **연도 비교가 아니다.**
   → 모든 산출물(표/그래프/문서)에 반드시 **"패치 15.1 vs 15.3 메타 드리프트"** 로 표기.
3. **큐 필터링**: 2024파일(`league_data`)은 큐가 섞여 있으므로 **솔로/듀오 랭크(queue_id = 420)** 만 남긴다.
4. **스키마 하모나이징**: 두 파일은 컬럼 스키마가 다르므로 **공통 스키마**로 변환 후 적재.
5. **밴 분석 금지**: 밴 데이터가 없으므로 밴률 분석은 하지 않는다. → **픽률·승률 기반**만.
6. **승부 예측 금지**: ML로 승패를 맞히는 모델은 만들지 않는다. 분석 방향이 어긋난다.

## 4. 기술 원칙 (Tech Principles)

- **적재/집계는 SQL 중심 (SQLite)** — 직무 필수 요건.
  - 단일 거대 테이블 지양. **정규화된 다중 테이블 + JOIN** 적극 활용.
  - 핵심 테이블 후보: `matches`, `participants`, `champions`, `players`, `patches`.
- **Python은 보조 역할** — 전처리 / K-means 클러스터링 / 시각화.
  - 라이브러리: `pandas`, `scikit-learn`, `matplotlib`, `seaborn`, `openpyxl`.
- **최종 대시보드는 Looker Studio** — 깨끗한 **집계 CSV**를 `outputs/`로 export 해서 연결.
- **Riot API 코드**는 `api/` 폴더에 격리. 키는 `.env`로 분리(레포에 절대 커밋 X).

## 5. 폴더 구조 (Folder Layout)

```
D:\프로젝트\game\
├── CLAUDE.md              # ← 이 문서
├── data/
│   ├── raw/               # 원본(2024.xlsx, 2025.csv) — 절대 수정 금지
│   └── processed/         # 하모나이즈 + 필터링 끝난 정형 데이터
├── sql/                   # 스키마(DDL) + 분석 쿼리(DML) — 번호 prefix로 실행순서 명시
├── src/                   # Python 스크립트(전처리/적재/클러스터링/플롯)
├── outputs/               # Looker Studio용 집계 CSV, 차트 PNG, 모델 아티팩트
├── docs/                  # 인사이트 메모(.md), 데이터 사전, 최종 README 초안
└── api/                   # Riot API 수집 파이프라인 (PHASE 5)
```

## 7. 커밋 규칙 (Commit Discipline)

각 **PHASE가 끝날 때마다** git commit 한다.

```
feat(phase-N): <한 줄 요약>
docs(phase-N): <한 줄 요약>
```

PR 형태가 아닌 1인 작업이므로 **PHASE 종료 = 커밋 시점**이 자연스러운 체크포인트.

## 8. 차별화 포인트 (Why this stands out)

- **"또 다른 LoL 승부 예측 노트북"이 아니다.** 라이브 게임 분석가가 실제 던지는 질문에 답한다.
  - 게임이 건강한가? (다양성 지표)
  - 유저는 어떻게 나뉘나? (행동 세그멘테이션)
  - 패치로 뭐가 바뀌었나? (드리프트)
- **SQL 중심 정규화 설계** — 직무 필수 요건을 정면으로 만족.
- **Riot API 직접 수집** — 정적 데이터셋 소비자가 아닌 데이터 엔지니어링 역량까지.
